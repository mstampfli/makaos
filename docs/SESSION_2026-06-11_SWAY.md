kernel better."*  Status at the end of the session:

## 1. The boot hang that wasn't a kernel bug

`FATAL: kernel read` from the UEFI loader after the kernel grew by
96 bytes: `mformat` had been formatting the ESP FAT to the **end of
the image** (266240-sector FAT inside a 2048-sector partition), so
file clusters past sector 4096 were silently clobbered by the ext2
`dd`.  kernel.bin's tail crossed that line.  build.sh now passes
`-T <partition sectors>`; an oversized kernel makes mcopy fail loudly.

## 2. Kernel fixes (commit `a944114`, plus follow-ups)

1. **Nested epoll** — epoll fds had `poll = NULL`, and the epoll_wait
   scan treats poll-less fds as always-ready.  libinput's context fd
   IS an epoll fd; wlroots' loop saw it permanently readable and
   busy-spun before its first frame (the historical "null-vtable"
   black screen — it was a NULL vfs-vtable *entry*).  Added
   `epoll_poll` (trylock + per-CPU depth cap) and the
   ep_poll_safewake-style wake cascade through the epoll file's waitq.
2. **DRM GETFB** minted-handle semantics (wlroots closes the creation
   handle right after ADDFB2; GETFB → ENOENT made the legacy cursor
   path drop EVERY commit → permanent black scanout).  Plus PRIME
   clone no longer shares/destroys the source's vgpu resource, and
   both clone paths charge the DRM quota.
3. **MSG_DONTWAIT** honored in sendmsg/recvmsg (libwayland keeps fds
   blocking and relies on per-call nonblock) + the SCM_RIGHTS drain
   uses a non-blocking recvfd (the blocking one parked dwl inside its
   first recvmsg forever while foot's hello sat unread).
4. **/dev/ptmx + /dev/pts/N** wired into sys_open/virtfs (park/claim
   model, TIOCGPTN).  The pty subsystem existed; the device nodes
   didn't — foot died with ENOENT on posix_openpt.
5. **mmap**: read-only MAP_PRIVATE on shmem fds (how clients map the
   compositor's xkb keymap fd).  **SYS_NPROC** for sysconf — the old
   -1 became foot's u16 worker count = 65535 threads.
6. **Signals after CPU migration** — the big one.  Redirect state
   moved off per-CPU cpu_t scratch onto the per-task kernel-stack
   frame (`SYSCALL_KFRAME` in process.h).  A task that slept
   mid-syscall and woke on another CPU read *another task's* user RSP:
   bash's SIGWINCH frame was built on a foot pthread stack and the
   kernel-mode write to the unmapped page was an unrecoverable #PF
   panic.  Entry asm keeps IRQs off until the per-task snapshot is
   complete; the exit-path signal branch is gone (setup_frame /
   sigreturn mutate the to-be-popped slots in place); fork and the
   arg5/6 snapshot read the kframe; the sigframe write is
   VMA-validated so a bad RSP kills the task, not the box.
7. **shmem cap 256MiB → 1GiB** (sparse; foot ftruncates its wl_shm
   scrollback memfd to 512MiB up front and aborts on failure).
8. **ext2 fast-path EOF clamp** (commit `303f87e`) — the aligned-read
   fast path rounded the remaining block count up and advanced the
   cursor past i_size; a 7447-byte file read back as 8192 bytes.
   sway's config loader counts bytes against st_size → "Config file
   changed during reading".
9. **Super/Meta scancodes** — E0-5B/5C/5D were absent from the
   extended scancode table, so no Mod4 binding could ever fire.

## 3. libc fixes (commits `2bc6165`, `e9e2ce5`)

- **getopt**: `reset_state()` zeroed `optarg` right after assignment —
  every `-x arg` consumer got NULL (dwl dropped `-s foot` silently;
  foot crashed on `-d info`).
- **execl** passed envp=0 → children lost WAYLAND_DISPLAY /
  XDG_RUNTIME_DIR ("failed to connect to wayland").
- **malloc/free**: no locking; two foot render threads could both
  pass the bump-allocator capacity check (PF-KILL above heap end).
  Spin+yield lock.  `heap_grow` also accepted brk()'s -errno returns
  as the new break (documented bug, was still live on this branch).
- **memfd_create** over shm_open+unlink (foot/wlroots otherwise back
  wl_shm pools with regular ext2 files; the huge sparse ftruncate
  fails).  MFD_*/F_SEAL_* in headers; seal fcntl is a soft no-op.
- **stdio**: `ftell` mixed a kernel-offset model into the
  consumer-position FILE (sway's detect_brace ftell/fseek pairs
  replayed the config stream mid-line); fseek SEEK_END probe re-syncs
  the kernel offset; getc/ungetc/freopen/popen/pclose/_unlocked
  implemented.
- **getcwd(NULL,0)** GNU auto-alloc (sway treats cwd-restore failure
  as a config error).
- **crt0 runs .init_array/.ctors** — statically-linked gobject
  bootstraps its type system from a constructor; without this every
  GLib consumer dies at first pango use (g_hash_table_lookup_node,
  CR2=0x38).  Linker script collects both sections into one window.
- **wordexp** (tilde + $VAR + field splitting; no command-sub, no
  glob), **atexit**, **mktime/difftime/ctime/asctime**, **wctomb**,
  **getaddrinfo** family, BIND4 resolver compat surface, utime/grp/
  statvfs/mntent/netinet-tcp/net-if/servent headers + stubs, and a
  header de-dup pass for ports compiled with -Werror=redundant-decls.

## 4. The sway port itself (commit `1eb2382`)

Dependency chain ported fresh: **fribidi 1.0.16** (drop -ansi),
**libpng 1.6.44**, **glib 2.82.2** (glib/gobject/gio/gmodule,
static), **cairo 1.18.2** (image+ft+png), **pango 1.54.0**, then
**sway 1.10.1** with `-Dswaybar=false -Dswaynag=false -Dtray=disabled
-Dgdk-pixbuf=disabled` (sd-bus has no port yet; swaybar/swaynag link
fine but were kept out of scope).  Notable port traps, all encoded in
the scripts:

- freetype2.pc must carry the **libtool** version (26.2.20) — cairo's
  `>= 9.7.3` check rejects the release-style 2.13.3.
- pixman-1.pc must tell the truth (0.42.2): overstating to 0.43.4
  made cairo enable `PIXMAN_r8g8b8_sRGB` against 0.42 headers.
- libinput's internal `list_*` symbols collide with sway's
  common/list.c under static linking → objcopy --redefine-sym in
  port-libinput.sh.
- sway's default config gets two lines stripped at image-install
  time (`include /etc/sway/config.d/*` — no glob in wordexp; the
  wallpaper line — PNG not installed; both are fatal config errors
  to sway).

## 6. Where it stands / next session's lead

`sway` on the image: full startup (DRM legacy modeset ✓, libinput ✓,
xkb keymap compile ✓, 228-line config clean ✓, workspace 1 ✓,
cursor theme ✓, swapchain + pixman buffers ✓, frame commits ✓),
Mod4+Return fires and `/bin/foot` is spawned and ELF-loaded.

**Blocker**: the spawned foot makes a handful of syscalls (fontconfig
opens) and then sits READY but never gets scheduled again, while
sway's glib worker thread burns a CPU in a `nanosleep` storm
(pthread_cond is a futex-less spin) and sway's main loop sits inside
`epoll_wait` in state RUNNING (wake/rescan churn).  Suspects, in
order:
1. Futex-less pthread_cond/mutex (spin + sched_yield) starving the
   work-stealing scheduler — a real kernel futex is the structural
   fix (also kills the CPU burn).
2. The new epoll wake cascade (`epoll_wake_func` → file_wq) possibly
   re-waking the outer loop in a storm — needs a wake-count probe.
3. `sched_wake`/steal interplay for READY-but-never-picked tasks
   (foot stayed READY for minutes on a 4-CPU box with ≤2 busy CPUs).

- Zombie reaping: sway's dead exec children stay zombies (SIGCHLD

- `uname`, `wmenu-run` missing on the image (cosmetic exec failures).
